### SKILL 1: 회의록 자동 정리

#### 파일명: `meeting-notes.md`

~~~markdown
---
name: 회의록 자동 정리

# 회의록 자동 정리 Skill

## 사용 시점
회의가 끝나고 메모를 정리할 때 사용합니다.

## 입력 형식
- 자유롭게 메모한 내용을 그대로 입력
- 날짜, 참석자 정보만 있으면 됨
- 불완전한 문장도 괜찮음

## 출력 형식

### 회의 정보
- 일시: [날짜] [시간]
- 장소: [회의실/온라인]
- 참석자: [이름(부서/직책)]
- 작성자: [이름]

### 안건
[회의 주제]

### 논의 내용
각 논의 사항을 명확한 문장으로 정리:
1. [주제 1]
   - [핵심 내용]
   - [제기된 의견]
   
2. [주제 2]
   - [핵심 내용]
   - [제기된 의견]

### 결정 사항
명확히 결정된 내용만 기록:
- [결정 1]
- [결정 2]

### 액션 아이템
담당자와 마감일을 명시:
- [ ] [담당자]: [해야 할 일] (마감: [날짜])
- [ ] [담당자]: [해야 할 일] (마감: [날짜])

### 다음 회의
- 일시: [예정일]
- 안건: [다음 논의 주제]

### 참고사항
회의 중 언급된 문서, 링크, 참고자료

## 작성 원칙
1. **명확성**: 애매한 표현 제거, 구체적으로
2. **간결성**: 불필요한 세부사항 생략
3. **실행 가능성**: 액션 아이템은 측정 가능하게
4. **중립성**: 개인 의견 배제, 사실만 기록

## 사용 예시

**입력:**
```
2026.1.7 오후 2시
홍길동, 김철수, 이영희
신규 회원 등급 시스템 논의
- 홍: 3단계 vs 4단계?
- 김: 개발 3주 걸림, 3단계가 빠름
- 이: 마케팅 관점에서 4단계가 좋음
- 결국 3단계로 시작하기로
- 홍길동이 다음주까지 기획서
- 수요일에 다시 미팅
```

**출력:**
```
회의록

일시: 2026년 1월 7일 (화) 14:00-15:00
참석자: 홍길동(기획팀), 김철수(개발팀), 이영희(마케팅팀)
작성자: [자동 생성]

안건
신규 회원 등급 시스템 설계 방향 결정

논의 내용
1. 등급 시스템 단계 수 결정
   - 3단계안: 구현 기간 3주, 빠른 출시 가능
   - 4단계안: 세밀한 혜택 제공, 마케팅 활용도 높음
   - 개발 리소스와 비즈니스 요구 간 균형 고려

결정 사항
- 1차로 3단계 등급 시스템 도입
- 추후 확장 가능하도록 설계
- 3개월 운영 후 데이터 기반 4단계 전환 재검토

액션 아이템
- [ ] 홍길동: 3단계 등급 시스템 상세 기획서 작성 (마감: 1/14)
- [ ] 김철수: 확장 가능한 시스템 아키텍처 검토
- [ ] 이영희: 각 등급별 혜택안 초안 작성

다음 회의
일시: 2026년 1월 15일 (수) 14:00
안건: 상세 기획서 검토 및 개발 일정 확정
```
~~~

~~~markdown
# 이메일 작성 도우미

## 사용 시점
- 프로젝트 관련 요청/보고
- 회의 일정 조율
- 진행 상황 공유
- 협업 요청

## 이메일 유형

### 1. 요청 이메일
**톤**: 정중하지만 명확하게

**구조**:
```
제목: [구체적인 요청 내용] 요청 드립니다

[수신자명] 님,

안녕하세요. [부서] [이름]입니다.

[요청 배경 1-2문장]

아래와 같이 [요청사항]을 요청 드립니다:
- [항목 1]
- [항목 2]
- [항목 3]

[마감일/일정]: [날짜]
[이유/근거]: [필요하다면]

검토 부탁드립니다.
감사합니다.

[이름]
```

### 2. 보고 이메일
**톤**: 간결하고 정확하게

**구조**:
```
제목: [프로젝트명] [기간] 진행 상황 보고

[수신자명] 님,

[프로젝트명]의 [기간] 진행 상황을 보고드립니다.

■ 완료 사항
- [항목 1]
- [항목 2]

■ 진행 중
- [항목 3] (진행률 X%)
- [항목 4] (진행률 X%)

■ 이슈
- [이슈 1]: [상황] → [대응 방안]

■ 향후 일정
- [날짜]: [계획]

추가 문의사항 있으시면 말씀해주세요.

[이름]
```

### 3. 회의 요청 이메일
**톤**: 명확하고 효율적으로

**구조**:
```
제목: [안건] 회의 요청

[수신자명] 님,

안녕하세요. [이름]입니다.

[안건]에 대해 논의가 필요하여 회의를 제안합니다.

■ 안건
[구체적인 논의 주제]

■ 참석자
[이름, 이름, 이름]

■ 소요 시간
약 [X분]

■ 가능한 일정 (아래 중 선택)
- [날짜] [시간]
- [날짜] [시간]
- [날짜] [시간]

회신 부탁드립니다.

[이름]
```

### 4. 피드백/수정 요청 이메일
**톤**: 건설적이고 구체적으로

**구조**:
```
제목: [문서명] 검토 및 피드백 요청

[수신자명] 님,

[문서명]을 첨부드립니다.

■ 주요 내용
- [핵심 1]
- [핵심 2]

■ 검토 요청 사항
1. [확인할 점 1]
2. [확인할 점 2]
3. [의견이 필요한 부분]

■ 피드백 마감
[날짜]까지 의견 주시면 감사하겠습니다.

[이름]
```

## 작성 원칙

### DO (해야 할 것)
- ✅ 제목에 핵심 내용 포함
- ✅ 첫 문장에 목적 명시
- ✅ 불릿 포인트로 가독성 확보
- ✅ 마감일/일정 명확히
- ✅ 액션 아이템 명확히

### DON'T (하지 말아야 할 것)
- ❌ 과도한 인사말
- ❌ 애매한 표현 ("가능하면", "아마도")
- ❌ 불필요한 배경 설명
- ❌ 여러 주제를 한 이메일에
- ❌ 긴급하지 않은데 "긴급" 표시

## 톤 조정

**격식 있게 (임원진, 외부)**
- "~입니다", "~드립니다"
- "검토 부탁드립니다"
- "감사합니다"

**일반적으로 (팀 내부)**
- "~입니다", "~해요"
- "확인 부탁해요"
- "감사합니다"

**캐주얼하게 (친한 동료)**
- "~해요", "~하려고요"
- "확인해줘"
- "고마워요"

## 사용 예시

**입력:**
"개발팀에 API 문서 요청하는 이메일 써줘. 다음주 금요일까지 필요해."

**출력:**
```
제목: 회원 등급 API 문서 요청

김철수 님,

안녕하세요. 기획팀 홍길동입니다.

회원 등급 시스템 기획을 진행 중이며, 
API 연동 설계를 위해 관련 문서가 필요합니다.

아래 API에 대한 문서를 요청 드립니다:
- 회원 정보 조회 API
- 등급 업데이트 API
- 구매 이력 조회 API

마감: 2026년 1월 17일 (금)

현재 문서가 없다면 간단한 명세서라도 
공유해주시면 감사하겠습니다.

홍길동
```
~~~

~~~markdown
# 상세 요구사항 정의서 작성

## 사용 시점
- 중대형 프로젝트 시작 전
- 외주 개발 발주 시
- 개발 범위 확정이 필요할 때

## 문서 구조 (20-30 페이지)

### 1. 문서 정보
```
프로젝트명: [프로젝트명]
문서 버전: v1.0
작성일: [날짜]
작성자: [기획팀 이름]
검토자: [개발 리더, PM]
승인자: [프로젝트 오너]

버전 이력:
v1.0 (2026.01.07): 초안 작성
```

### 2. 프로젝트 개요

**2.1 프로젝트 배경**
- 현재 상황
- 문제점/기회
- 해결 방향

**2.2 프로젝트 목적**
- 비즈니스 목표
  - [정량 목표]: 매출 X% 증가
  - [정량 목표]: 사용자 Y명 확보
- 기술적 목표
  - [목표 1]
  - [목표 2]

**2.3 프로젝트 범위**

포함 사항:
- [기능 1]

**[F-001] 회원가입**

개요:
신규 사용자가 이메일과 비밀번호로 계정을 생성한다.

입력:
- 이메일 (필수): 유효한 이메일 형식

4. 가입 완료 화면 표시

- 성공: 가입 완료 페이지 + 인증 이메일 발송
~~~

~~~markdown
### 시나리오: 김지현의 급한 선물 구매

**페르소나**: 김지현 (34세, 마케팅 매니저)
- 기술 활용도 높음
- 시간 효율 중시
- 품질보다 빠른 배송 선호

**목표**: 친구 생일 선물을 오늘 안에 주문하기

**상황**: 
- 시간: 오후 6시 (퇴근 직전)
- 장소: 회사 사무실
- 맥락: 내일이 친구 생일, 깜빡 잊고 있다가 방금 생각남

---

**시작 상태**
김지현은 회사 컴퓨터를 끄기 직전, 내일이 친구 생일임을 갑자기 기억해낸다. "아, 맞다! 지혜 생일이 내일인데..."라며 당황한다. 😰 퇴근 후 직접 가게에 갈 시간은 없고, 최소한 내일 아침까지는 배송이 되어야 한다. 급하게 스마트폰을 꺼내 쇼핑앱을 연다.

**단계별 여정**

🎬 **Step 1: 앱 실행 및 목표 인지**
- **행동**: 쇼핑앱 아이콘을 탭하여 실행
- **생각**: "선물용으로 괜찮은 거... 30만원 정도면 되겠지?"
- **감정**: 급함 😰 + 약간 불안함
- **기대**: 빠르게 선물용 상품을 찾고 싶다

💡 **시스템 반응**:
- 홈 화면 로딩 (0.5초)
- 최근 본 상품(주방용품)이 먼저 보임

✅ **결과**:
지현은 최근 본 상품이 선물용이 아니라는 것을 인지하고 검색으로 전환 결정

---

🎬 **Step 2: 검색 및 필터 설정**
- **행동**: 상단 검색바를 탭하고 "여성 향수"를 입력
- **생각**: "향수면 무난하겠지. 근데 내일 아침까지 올까?"
- **감정**: 조금 진정 🤔
- **기대**: 내일 배송 가능한 상품만 보고 싶다

💡 **시스템 반응**:
- 검색 결과 50개 표시
- 정렬 옵션: "인기순" 기본값
- 필터: 가격/브랜드/배송 등

✅ **결과**:
많은 결과에 압도됨. "내일 배송" 필터를 찾기 시작

---

🎬 **Step 3: 배송 필터 적용**
- **행동**: 필터 아이콘 탭 → "내일 도착" 옵션 선택
- **생각**: "오, 이렇게 필터하면 되겠네"
- **감정**: 약간 안심 😊
- **기대**: 확실히 내일 받을 수 있는 상품만 나오길

💡 **시스템 반응**:
- 결과 50개 → 8개로 줄어듦
- 각 상품 카드에 "내일 아침 도착" 배지 표시

✅ **결과**:
선택지가 줄어들어 오히려 편함. 상품을 하나씩 살펴보기 시작

---

🎬 **Step 4: 상품 비교 및 선택**
- **행동**: 3개 상품의 상세페이지를 번갈아가며 확인
- **생각**: "A는 리뷰가 좋네. B는 포장이 예쁘고. C는..."
- **감정**: 선택 고민 🤔
- **기대**: 실패하지 않을 선택을 하고 싶다

💡 **시스템 반응**:
- 각 상품의 상세 정보 표시
- 리뷰 (평점, 포토 리뷰)
- "선물용 포장" 옵션 표시

✅ **결과**:
"디올 미스 디올" 선택 (리뷰 4.8점, 선물 포장 무료)

---

🎬 **Step 5: 주문 및 결제**
- **행동**: "바로 구매" 버튼 탭
- **생각**: "배송지는 회사로... 내일 아침에 받아서 포장해서 가져가야지"
- **감정**: 안심 + 서두름 😌
- **기대**: 빠르게 결제 완료하고 싶다

💡 **시스템 반응**:
- 저장된 배송지 목록 표시 (집/회사)
- 저장된 결제수단 목록 (카드 3개)
- "선물용 포장" 자동 체크됨

✅ **결과**:
회사 주소 선택, 기본 카드로 결제, 총 3탭만에 완료

---

**분기: 재고 부족 시**

만약 선택한 상품의 재고가 없다면:

1. **사용자 반응**: "에이, 진짜?" 😤 다시 검색으로
2. **시스템 대응**: "이 상품은 품절입니다. 비슷한 상품을 추천해드릴까요?" 팝업
3. **대안 경로**:
   - 추천 상품 3개 표시 (같은 브랜드, 비슷한 가격대)
   - "내일 배송" 보장되는 상품만
   - 한 번의 탭으로 바로 구매 페이지로

---

**최종 결과**

**성공 시**:
- 주문 완료 (소요 시간: 5분)
- 사용자 만족도: ⭐⭐⭐⭐ (4/5)
  - 좋았던 점: 빠른 필터, 저장된 정보 활용
  - 아쉬운 점: 초기에 너무 많은 결과
- 다음 행동: 친구에게 "내일 선물 줄게" 메시지 보냄

**실패 시** (재고 없는 경우):
- 사용자 감정: 짜증 😤 → 경쟁 앱으로 이동 가능성
- 시스템 개선점: 
  - 재고 실시간 표시 필요
  - 대안 상품 추천 강화
  - "알림 신청" 기능 추가

---

**인사이트**

이 시나리오에서 발견된 것:

1. **시간 압박이 있는 사용자는 필터를 적극 활용한다**
   → "내일 배송" 같은 명확한 필터가 중요

2. **선택지가 너무 많으면 오히려 스트레스**
   → 필터 적용 후 10개 이하로 줄이는 것이 이상적

3. **저장된 정보(주소, 카드)가 구매 속도를 크게 높인다**
   → 첫 구매 시 정보 저장 유도 필요

4. **선물용 니즈에 특화된 기능이 차별화 포인트**
   → 선물 포장, 메시지 카드, 배송 시간 지정 등

5. **실패 경험(품절)의 대응이 이탈을 막는다**
   → 대안 제시가 핵심
~~~

~~~markdown
   - 권장: Q2 로드맵 포함

   - 발견: 경쟁사들은 일방향 소통만
   - 기회: 사용자 충성도 향상
   - 필요: 커뮤니티 관리 인력
   - 권장: 베타 버전으로 시작
~~~

~~~markdown
# UX Writing 가이드

## 사용 시점
- 버튼, 레이블, 메시지 등 모든 UI 텍스트 작성
- 에러 메시지, 안내 문구 작성
- 온보딩, 튜토리얼 문구 작성

## UX Writing 원칙

### 1. 명확성 (Clarity)
창의성보다 이해 가능성이 우선

❌ "당신만의 특별한 여정을 시작하세요"
✅ "회원가입 하기"

### 2. 간결성 (Conciseness)
모바일 화면에서 읽기 쉽게

❌ "지금 바로 가입하시면 다양한 혜택을 받으실 수 있습니다"
✅ "가입하고 혜택 받기"

### 3. 일관성 (Consistency)
같은 의미는 같은 단어로

❌ "취소", "닫기", "그만두기"를 섞어 쓰기
✅ "취소"로 통일

### 4. 대화체 (Conversational)
친근하지만 가볍지 않게

❌ "오류가 발생하였사오니 다시 시도하여 주시기 바랍니다"
✅ "문제가 발생했어요. 다시 시도해주세요"

### 5. 긍정적 (Positive)
부정보다 긍정 표현

❌ "비밀번호를 틀렸습니다"
✅ "비밀번호를 다시 확인해주세요"

## 상황별 작성 가이드

### 버튼 (CTA - Call To Action)

**원칙**:
- 동사로 시작
- 구체적인 행동 명시
- 2-4단어

**예시**:

주문/결제:
- ✅ "구매하기" (간결)
- ✅ "35,000원 결제하기" (금액 명시로 확신 제공)
- ❌ "클릭" (무엇을 하는지 불명확)

저장/완료:
- ✅ "저장" 
- ✅ "변경사항 저장"
- ❌ "OK" (모호함)

취소/삭제:
- ✅ "취소"
- ✅ "삭제하기"
- ⚠️ "삭제" + 확인 단계 추가 (중요 행동)

### 에러 메시지

**구조**: [무엇이 문제인가] + [어떻게 해결하는가]

**좋은 예시**:

```
❌ "Error 404"
✅ "페이지를 찾을 수 없어요
    링크를 다시 확인해주세요"

❌ "Invalid email format"
✅ "이메일 형식이 올바르지 않아요
    example@email.com 형식으로 입력해주세요"

❌ "Network error"
✅ "인터넷 연결을 확인해주세요
    Wi-Fi나 데이터가 켜져 있나요?"
```

**에러 메시지 템플릿**:

```
[감정 표현] [문제 설명]
[구체적 해결 방법]
[대안 제시]
```

예시:
```
죄송해요, 결제가 완료되지 않았어요

카드 정보를 다시 확인해주시거나
다른 결제 수단을 선택해주세요

[다시 시도] [고객센터 문의]
```

### 성공 메시지

**톤**: 축하하지만 과하지 않게

```
✅ "저장되었습니다" (간단명료)
✅ "주문이 완료되었어요! 
    배송은 내일 도착 예정이에요" (추가 정보)
❌ "와! 대단해요! 축하합니다!" (과함)
```

### 빈 상태 (Empty State)

**구조**: [상황 설명] + [액션 유도]

```
[아이콘]
아직 담은 상품이 없어요
마음에 드는 상품을 찾아볼까요?

[상품 둘러보기]
```

```
[아이콘]
아직 알림이 없어요
새로운 알림이 오면 여기서 확인할 수 있어요
```

### 온보딩 메시지

**원칙**:
- 단계당 1개 메시지
- 가치 중심 (기능 나열 X)
- 3-5단계 이내

**좋은 예시**:

Step 1:
```
원하는 상품을 빠르게 찾아요
AI 검색으로 이미지만 찍어도 OK
```

Step 2:
```
매일 새로운 혜택을 받아요
등급별 맞춤 할인과 쿠폰
```

Step 3:
```
안전하게 결제해요
간편결제로 클릭 3번이면 끝
```

### 확인 요청 (Confirmation)

중요한 액션 전에 한 번 더 확인

```
정말 삭제할까요?
삭제하면 복구할 수 없어요

[취소]  [삭제하기]
```

**버튼 순서 규칙**:
- iOS: [취소] [확인]
- Android: [확인] [취소]
- 웹: 일반적으로 [취소] [확인]

### 로딩 메시지

**짧은 로딩 (1-3초)**:
- 메시지 없이 스피너만

**중간 로딩 (3-10초)**:
```
상품을 불러오는 중이에요...
```

**긴 로딩 (10초 이상)**:
```
첫 설정을 준비하고 있어요
잠시만 기다려주세요 (약 30초 소요)
```

**팁**: 진행률 표시 추가

## 톤앤매너 (Tone & Manner)

### 우리 서비스의 목소리

**성격**: [친근한 / 전문적인 / 트렌디한 등]
**말투**: [존댓말 / 반말 / 혼합]

**DO (해야 할 것)**:
- ✅ 친구에게 말하듯 편안하게
- ✅ 사용자를 존중하는 태도
- ✅ 이모지 적절히 사용 (🎉 ✅ ❤️ 등)

**DON'T (하지 말아야 할 것)**:
- ❌ 기계적인 말투
- ❌ 과도한 존댓말
- ❌ 유치한 표현
- ❌ 강요하는 톤

```

**중요한 안내**: 명확하고 진지하게

결제가 완료되었습니다
주문 번호: #123456
~~~

~~~markdown
# 정보구조(IA) 설계 Skill

## 사용 시점
- 신규 서비스 구축 시
- 대규모 리뉴얼 시
- 콘텐츠가 많아져 정리가 필요할 때

## IA 설계 프로세스

### 1단계: 콘텐츠 인벤토리
모든 콘텐츠와 기능을 나열

```
ID | 콘텐츠/기능 | 현재 위치 | 중요도 | 접근 빈도
---|-------------|-----------|--------|----------
001 | 상품 검색 | 홈 상단 | 높음 | 매우 높음
002 | 카테고리 목록 | 메뉴 | 높음 | 높음
003 | 마이페이지 | 하단 탭 | 중간 | 중간
...
```

### 2단계: 사용자 니즈 매핑
사용자가 찾는 정보/기능 우선순위

```
페르소나 A (신규 사용자):
1. 상품 둘러보기 (탐색)
2. 인기 상품 보기
3. 회원가입

페르소나 B (기존 사용자):
1. 검색
2. 재구매
3. 주문 확인
```

### 3단계: 카드 소팅
사용자가 자연스럽게 생각하는 그룹핑 반영

```
그룹 1: 쇼핑
- 상품 검색
- 카테고리
- 추천 상품
- 이벤트

그룹 2: 내 정보
- 주문 내역
- 찜 목록
- 쿠폰
- 포인트

그룹 3: 고객 지원
- FAQ
- 1:1 문의
- 공지사항
```

### 4단계: 정보 계층 구조 설계

```
Level 1 (최상위)
├─ 홈
├─ 카테고리
├─ 검색
├─ 찜
└─ MY

Level 2 (홈 하위)
홈
├─ 메인 배너
├─ 카테고리 바로가기
├─ 추천 상품
├─ 이벤트
└─ 최근 본 상품

Level 2 (카테고리 하위)
카테고리
├─ 의류
│   ├─ 여성 의류
│   │   ├─ 상의
│   │   ├─ 하의
│   │   └─ 아우터
│   └─ 남성 의류
├─ 뷰티
└─ 리빙

Level 2 (MY 하위)
MY
├─ 주문 관리
│   ├─ 주문/배송 조회
│   ├─ 반품/교환
│   └─ 취소 내역
├─ 혜택 관리
│   ├─ 쿠폰
│   └─ 포인트
└─ 설정
    ├─ 회원 정보
    └─ 알림 설정
```

### 5단계: 네비게이션 디자인

**주 네비게이션 (Primary Navigation)**:
- 위치: 하단 탭 바
- 항목 수: 5개 (최대)
- 항목: 홈, 카테고리, 검색, 찜, MY

**보조 네비게이션 (Secondary Navigation)**:
- 위치: 상단 헤더
- 항목: 장바구니, 알림

**유틸리티 네비게이션 (Utility Navigation)**:
- 위치: 햄버거 메뉴 또는 MY 내부
- 항목: 설정, 고객센터, 이벤트 등

### 6단계: 깊이 최적화

**원칙**: 3클릭 룰
중요한 콘텐츠는 홈에서 3클릭 이내 도달

**현재 깊이 분석**:
```
상품 상세 페이지 도달:
나쁜 예: 홈 → 카테고리 → 대분류 → 중분류 → 소분류 → 상품 (5클릭)
좋은 예: 홈 → 검색 → 상품 (2클릭)
        홈 → 추천 섹션 → 상품 (2클릭)
```

### 7단계: IA 다이어그램 작성

**시각화 도구**: Mermaid, Figma, Whimsical

graph TD

    A --> C[검색]
    A --> D[찜]
    A --> E[MY]

    B --> B2[뷰티]
    B --> B3[리빙]
    
    B1 --> B1a[여성 의류]

    E --> E1[주문 관리]
    E --> E2[혜택 관리]
    E --> E3[설정]
    
    E1 --> E1a[주문/배송 조회]
    E1 --> E1b[반품/교환]
```

## IA 평가 기준

### 1. 발견 가능성 (Findability)
사용자가 원하는 것을 쉽게 찾는가?

테스트: "상품 반품하기"를 5초 안에 찾을 수 있는가?

### 2. 명확성 (Clarity)
메뉴 이름이 명확한가?

❌ "서비스" (뭐가 있는지 불명확)
✅ "고객 지원" (명확)

### 3. 일관성 (Consistency)
같은 레벨은 같은 방식으로 접근

✅ 모든 하위 페이지에서 "뒤로 가기" 버튼이 같은 위치

### 4. 확장성 (Scalability)
새 콘텐츠를 쉽게 추가할 수 있는가?

좋은 구조: 카테고리 추가 시 큰 변경 없음

## 사용자 테스트

### 트리 테스트 (Tree Test)
사용자에게 텍스트 기반 메뉴만 보여주고 특정 항목을 찾게 함

```
Task: "이번 주 배송 예정인 상품을 확인하세요"

메뉴:
1. 홈
2. 카테고리
3. MY
   3-1. 주문 관리
        3-1-1. 주문/배송 조회 ← (정답)
        3-1-2. 반품/교환
   3-2. 혜택 관리
   3-3. 설정
4. 고객 지원

성공률: 85% (목표: 80% 이상)
~~~

~~~markdown
# A/B 테스트 설계 Skill

## 사용 시점
- 새 기능의 효과를 검증하고 싶을 때
- 디자인 개선의 영향을 측정하고 싶을 때
- 두 가지 옵션 중 선택이 어려울 때

## A/B 테스트 설계 프로세스

### 1단계: 가설 수립

**SMART 가설**:
- Specific (구체적)
- Measurable (측정 가능)
- Achievable (달성 가능)
- Relevant (관련성 있음)
- Time-bound (기한 설정)

**나쁜 가설**:
❌ "버튼을 크게 하면 좋을 것 같다"

**좋은 가설**:
✅ "CTA 버튼의 크기를 40px에서 48px로 키우면, 
    결제 전환율이 현재 15%에서 18%로 3%p 상승할 것이다"

### 2단계: 메트릭 정의

**Primary Metric (주요 지표)**:
- 이 테스트의 성공을 판단하는 핵심 지표
- 예: 전환율, 클릭률, 구매 완료율

**Secondary Metrics (보조 지표)**:
- 부수적인 효과를 측정
- 예: 평균 체류 시간, 이탈률, 평균 주문 금액

**Counter Metrics (역효과 지표)**:
- 개선이 부작용을 일으키지 않는지 확인
- 예: 로딩 시간, 서버 부하, 고객 문의 증가

### 3단계: 변형안 설계

**Control (대조군 - A)**:
현재 버전 (변경 없음)

**Variant (실험군 - B)**:
변경 사항 적용

**중요**: 한 번에 하나만 변경
- ✅ 버튼 색상만 변경
- ❌ 버튼 색상 + 크기 + 위치 모두 변경 (무엇이 효과인지 불명확)

### 4단계: 샘플 크기 계산

**필요한 정보**:
- 현재 전환율: 15%
- 목표 전환율: 18%
- 통계적 유의수준: 95%
- 검정력: 80%

**계산 결과**:
각 그룹당 약 2,000명 필요
→ 총 4,000명

**예상 기간**:
일 방문자 500명 → 8일 소요

### 5단계: 트래픽 분배

**일반적인 분배**:
- A (Control): 50%
- B (Variant): 50%

**조심스러운 테스트** (리스크 큰 변경):
- A: 90%
- B: 10%

**다변량 테스트** (3개 이상):
- A: 33%
- B: 33%
- C: 34%

## A/B 테스트 문서 템플릿

```markdown
# A/B 테스트 계획서

## 기본 정보

**테스트명**: [간단명료한 이름]
**담당자**: [이름]
**일정**: 2026.01.10 ~ 2026.01.20 (10일)
**상태**: 계획 중 / 실행 중 / 완료

## 1. 배경 및 목적

**현재 상황**:
결제 페이지의 전환율이 15%로, 업계 평균 20%보다 낮음

**문제**:
사용자 피드백에서 "버튼이 눈에 안 띈다"는 의견 다수

**목적**:
CTA 버튼의 가시성을 높여 결제 전환율 향상

## 2. 가설

**가설 1** (Primary):
CTA 버튼 크기를 키우면 클릭률이 증가하여 전환율이 상승할 것이다

**가설 2** (Secondary):
버튼이 커지면 시각적으로 더 눈에 띄어 결제 페이지 체류 시간이 감소할 것이다

## 3. 메트릭

### Primary Metric
- **전환율 (Conversion Rate)**
  - 정의: (결제 완료 / 결제 페이지 방문) × 100
  - 현재 값: 15%
  - 목표 값: 18% (3%p 증가)
  - 측정 도구: Google Analytics 4

### Secondary Metrics
- **클릭률 (CTR)**
  - 정의: (버튼 클릭 / 페이지 조회) × 100
  - 현재 값: 45%
  - 목표 값: 55%

- **평균 결제 시간**
  - 정의: 페이지 진입부터 완료까지 소요 시간
  - 현재 값: 180초
  - 목표 값: 150초

### Counter Metrics
- **이탈률 (Bounce Rate)**
  - 현재 값: 30%
  - 허용 범위: 35% 이하 (5%p 증가까지는 허용)

- **평균 주문 금액 (AOV)**
  - 현재 값: 50,000원
  - 허용 범위: 45,000원 이상 (10% 감소까지 허용)

## 4. 변형안 설계

### A안 (Control - 대조군)

**현재 버전**:
- 버튼 크기: 40px 높이
- 버튼 색상: #0066FF (파란색)
- 버튼 텍스트: "결제하기"
- 위치: 하단 고정

[스크린샷 또는 디자인 시안]

### B안 (Variant - 실험군)

**변경 사항**:
- 버튼 크기: 48px 높이 (20% 증가) ← 유일한 변경점
- 버튼 색상: #0066FF (동일)
- 버튼 텍스트: "결제하기" (동일)
- 위치: 하단 고정 (동일)

[스크린샷 또는 디자인 시안]

**변경 이유**:
모바일 환경에서 손가락 터치 영역을 고려한 최소 권장 크기 48px 적용

## 5. 실험 설계

### 샘플 크기
- 최소 필요 샘플: 각 그룹당 2,000명
- 예상 일 방문자: 500명
- 예상 테스트 기간: 8일
- 실제 테스트 기간: 10일 (버퍼 포함)

### 트래픽 분배
- A안 (Control): 50%
- B안 (Variant): 50%

### 랜덤화 방법

- 쿠키 유효 기간: 30일

## 6. 성공 기준

### 최소 성공 기준 (Must Have)

- [ ] 통계적 유의성 확보 (p < 0.05)

- [ ] 클릭률 10%p 이상 증가

| 샘플 부족 (휴일 등) | 중 | 중 | 테스트 기간 연장 |
| 기술적 오류 | 낮 | 상 | 개발팀 대기, 즉시 롤백 가능하도록 |

## 9. 분석 계획

### 세그먼트 분석
전체 결과 외에 다음 세그먼트별로도 분석:
- 신규 vs 재방문 사용자
- 연령대별 (20대 / 30대 / 40대+)
- 디바이스별 (iOS / Android)

### 예상 시나리오

**시나리오 1**: B안 승리 (전환율 18%)
→ 즉시 전체 롤아웃

**시나리오 2**: 차이 없음 (전환율 15% 유지)
→ 다른 요소 테스트 (색상, 위치 등)

**시나리오 3**: A안 승리 (B안 하락)
→ 가설 재검토, 사용자 인터뷰

## 10. 결과 리포트 (테스트 후 작성)

### 요약
- [승자]: B안
- [핵심 메트릭 변화]: 전환율 15% → 17.5% (2.5%p 증가)
- [통계적 유의성]: p = 0.023 (유의미)
- [결정]: B안 전체 적용

### 상세 데이터

| 메트릭 | A안 | B안 | 차이 | p-value |
|--------|-----|-----|------|---------|
| 전환율 | 15.0% | 17.5% | +2.5%p | 0.023 |
| 클릭률 | 45% | 53% | +8%p | 0.011 |
| 평균 결제 시간 | 180초 | 165초 | -15초 | 0.041 |
| 이탈률 | 30% | 31% | +1%p | 0.234 (유의 X) |

### 세그먼트 인사이트

**재방문 사용자**:
- 효과 큼 (전환율 18% → 21%)
- 이미 프로세스에 익숙해서 버튼만 키워도 효과

**신규 사용자**:
- 효과 보통 (전환율 12% → 14%)
- 다른 개선 필요 (안내 문구, 튜토리얼 등)

### 학습 및 다음 액션

**학습한 것**:
- 버튼 크기는 전환율에 영향을 줌
- 특히 재방문 사용자에게 효과적
- Counter Metric에 부정적 영향 없음

**다음 테스트**:
- 신규 사용자를 위한 온보딩 개선
- 버튼 색상 테스트 (빨간색 vs 파란색)
~~~

~~~markdown
## 레이아웃 원칙

### 1. 여백 활용

### 3. 정렬
- 모든 요소를 정렬선에 맞추기
- 좌/우, 상/하 정렬 일관성
- 중앙 정렬은 제목에만

## 색상 사용

### 메인 색상
- Primary: 브랜드 색상 (로고, 강조)
- Accent: 포인트 색상 (차트, 아이콘)
- Neutral: 회색 계열 (텍스트, 배경)

### 색상 규칙
- 슬라이드당 최대 3-4가지 색상
- 배경과 텍스트는 충분한 대비
- 색맹 고려 (빨강-초록 조합 피하기)

## 타이포그래피

### 폰트 선택
- 제목: 고딕체, Bold
- 본문: 고딕체, Regular
- 숫자: 산세리프체

### 크기
- 제목: 32-40pt
- 부제: 24-28pt
- 본문: 18-22pt
- 캡션: 14-16pt

### 가독성
- 한 줄에 최대 10-12단어
- 줄간격: 1.2-1.5배
- 텍스트 블록은 좌측 정렬

## 이미지 사용

### 선택 기준
- 고해상도 (최소 1920x1080)

### 배치
- 텍스트 위에 이미지 올릴 때: 어두운 오버레이 추가

- 여러 이미지: 같은 크기/스타일로 통일
~~~

~~~markdown
- 드롭다운 메뉴

### Level 3: 데이터 시뮬레이션
- 로컬 스토리지 사용
- 가짜 데이터로 CRUD
- 간단한 상태 관리
~~~

**Claude.ai와 Claude Code에서 PPT를 만드는 방식**

Claude.ai와 Claude Code에서 PPT를 만드는 방식은 크게 두 가지로 나뉩니다. **(1) 웹 서비스(Claude.ai)의 내장 기능**을 사용하는 방식과 **(2) Claude Code(CLI) 및 Agent Skills API**를 통해 코드로 제어하는 방식입니다.

## 1. Claude.ai (웹/데스크탑 앱) 사용 시

Claude.ai의 **'Upgraded file creation'** (실험실 기능)을 활용하는 방법입니다.

* **양식 정의**: 사용자가 평소 사용하는 **기업 표준 PPT 템플릿(`.pptx`) 파일** 자체가 정의서가 됩니다.
* **호출 및 사용 방법**:
1. **설정**: `Settings` -> `Features` -> `Experimental`에서 **"Upgraded file creation and analysis"**를 활성화합니다.
2. **업로드**: 채팅창에 자신의 PPT 템플릿 파일을 업로드합니다.
3. **호출**: "첨부된 템플릿의 디자인과 레이아웃을 유지하면서 [주제]에 대한 발표 자료를 만들어줘"라고 요청합니다.
4. **결과**: Claude가 내부적으로 Python 코드를 실행하여 템플릿의 슬라이드 마스터와 스타일을 분석한 뒤, 그 양식에 맞춰 내용을 채운 새 PPT 파일을 생성해 줍니다.



---

## 2. Claude Code 및 Agent Skills (개발자용)

터미널 기반의 Claude Code나 API를 사용할 때는 **'Skill'**이라는 단위로 기능을 호출합니다.

* **양식 정의**:
* **SKILL.md**: 기술의 사용법과 워크플로우를 정의하는 문서입니다. 여기에 "템플릿의 특정 인덱스 슬라이드를 복제하라"는 식의 로직을 기술합니다.
* **Python 스크립트**: 실제 PPT를 조작하는 `python-pptx` 라이브러리 기반의 코드를 작성하여 양식을 제어합니다.


* **호출 및 사용 방법**:
* **내장 Skill 호출**: Claude Code에서 별도의 설정 없이 "Create a presentation from template.pptx"라고 명령하면, Anthropic이 제공하는 기본 `pptx` Skill이 실행됩니다.
* **사용자 정의 Skill**: `public/pptx/SKILL.md`와 같은 경로에 자신만의 템플릿 처리 로직을 담은 Skill을 생성하고, Claude에게 "이 Skill을 사용해서 PPT를 만들어"라고 지시할 수 있습니다.
