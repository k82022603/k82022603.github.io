## 들어가며: "생각하는 모든 게 다 된다"는 말의 의미

소셜 미디어에 심심찮게 올라오는 클로드 코드 사용 후기 중 가장 자주 등장하는 문장이 있다. "정말 안 되는 게 없다. 생각하는 모든 게 다 된다." 처음 이 문장을 접했을 때는 과장처럼 들린다. 하지만 Claude Skills라는 시스템의 구조와 작동 원리를 이해하고 나면, 이것이 과장이 아니라 사실에 가까운 진술임을 알게 된다.

핵심은 하나다. AI가 강력해졌을 때 그 능력을 최대로 끌어내는 방법은 모델 자체를 바꾸는 것이 아니라, 그 모델이 특정 작업을 수행할 때 필요한 **맥락과 지식을 얼마나 정밀하게 사전에 설계해 두느냐**에 달려 있다. Skills는 그 설계의 그릇이다.

앤트로픽은 2025년 10월 Agent Skills를 공식 출시하며 이를 **"신입 직원 온보딩 가이드를 작성하는 것과 같다"** 고 설명했다. 전문 지식을 폴더 구조로 패키징하면 Claude가 이를 동적으로 로드하여 활용한다는 뜻이다. 이 비유가 정확한 이유는, Skills가 단순히 코드를 실행하는 스크립트가 아니라 Claude가 **어떻게 생각하고 행동해야 하는지를 가르치는 교육 문서**이기 때문이다.

## 1. Skills의 구조를 이해해야 활용이 열린다

### SKILL.md: 모든 것의 시작

Skills의 핵심 파일은 `SKILL.md`다. 이 파일은 두 부분으로 구성된다. 파일 상단의 YAML frontmatter 영역에는 스킬의 이름(`name`), 설명(`description`), 사용 도구 제한(`allowed-tools`) 같은 메타데이터를 작성한다. 그 아래 마크다운 본문에는 Claude가 이 스킬을 실행할 때 따라야 하는 구체적인 지침, 워크플로우, 참고 자료들을 자연어로 기술한다.

중요한 것은 Claude가 스킬을 **호출할 타이밍을 스스로 판단한다**는 점이다. `description` 필드가 바로 그 판단 기준이다. Claude는 대화 속에서 사용자의 의도를 파악하고, 현재 상황에 어떤 스킬이 가장 적합한지를 description에 기반하여 자동으로 결정한다. 따라서 description을 얼마나 정교하게 쓰느냐가 스킬의 자동 트리거 품질을 결정한다.

### 3단계 로딩 시스템: 맥락을 낭비 없이 쓴다

Skills는 컨텍스트 윈도우를 효율적으로 관리하기 위해 3단계 로딩 방식을 채택하고 있다. 1단계에서 메타데이터(name + description)는 세션 시작 시 항상 컨텍스트에 로드된다. 2단계인 SKILL.md 본문은 스킬이 트리거될 때만 로드된다. 3단계인 번들 리소스(스크립트, 템플릿, 자산 파일 등)는 Claude가 필요하다고 판단할 때만 로드된다. 이 구조 덕분에 수십 개의 스킬을 등록해두어도 실제로 사용되는 스킬의 컨텍스트만 소비되어 낭비가 없다.

### 폴더 구조: 스킬은 단순한 파일이 아니다

하나의 스킬 폴더는 다음과 같은 구성을 가진다.

```
my-skill/
├── SKILL.md        # 핵심 지침 파일
├── scripts/        # Claude가 실행할 수 있는 셸 스크립트, Python 등
├── templates/      # 출력에 활용할 템플릿 파일
└── assets/         # 참조 이미지, 아이콘, 보일러플레이트 등
```

스크립트를 스킬 폴더 안에 함께 패키징할 수 있기 때문에, Claude는 단순히 지시를 내리는 것을 넘어 실제로 코드를 실행하고 파일 시스템과 상호작용하며 외부 서비스를 호출하는 **에이전트 수준의 작업**을 수행할 수 있게 된다.

### 호출 방식: 자동 vs 수동

스킬에는 두 가지 호출 방식이 있다. 기본적으로 Claude가 컨텍스트를 보고 자동으로 로드하는 **모델 자동 호출** 방식이 있고, 사용자가 `/skill-name` 형태로 명시적으로 트리거하는 **수동 호출** 방식이 있다. `disable-model-invocation: true` 옵션을 frontmatter에 추가하면 해당 스킬은 반드시 사용자가 직접 명령해야만 실행된다. 배포, 프로덕션 환경 조작처럼 신중하게 제어해야 하는 작업에 이 옵션을 활용한다.

## 2. SNS 자동화: 영상 편집부터 이미지 크롭까지

SNS 자동화는 Skills의 가장 실용적인 활용 영역 중 하나다. 표면적으로는 "포스팅을 자동화한다"는 단순한 이야기처럼 들리지만, 실제 구현 범위는 훨씬 넓다.

블로그 글 하나를 입력하면 다섯 장짜리 인스타그램 카드 뉴스로 변환하고, 각 카드에 핵심 포인트를 한 가지씩 담되 마지막 카드에 CTA를 배치하는 자동화가 실제로 작동한다. 이미지의 비율이 SNS 플랫폼별 권장 규격에 맞지 않을 경우 자동으로 크롭하거나 리사이징하는 처리도 스킬 안에 스크립트로 구현할 수 있다. 브랜드 가이드라인을 assets 폴더에 참조 문서로 넣어두면, 색상 팔레트나 폰트 규칙을 지키며 이미지를 생성하는 일도 가능하다.

이 모든 것이 단일 스킬 안에 패키징될 수 있다. SNS 마케팅 이미지 스킬을 예로 들면, `SKILL.md`에 브랜드 톤앤매너와 플랫폼별 가이드라인을 기술하고, `scripts/` 폴더에 이미지 크롭 스크립트와 포스팅 API 호출 코드를 넣어두면 된다. Claude는 이 지침을 내재화해 적절한 도구와 스크립트를 선택하고, 사용자가 "인스타그램용으로 변환해줘"라고 말하는 순간 전체 파이프라인을 자율적으로 실행한다.

영상 편집의 경우에도 Remotion 스킬을 활용하면 자연어 지시만으로 React 기반의 프로그래매틱 비디오를 생성할 수 있다. 데모 영상, 제품 소개 클립, SNS 릴스 형식의 짧은 영상을 별도의 영상 편집 도구 없이 만들어낼 수 있다는 뜻이다.

## 3. agent-browser: SNS 자동화의 핵심 도구

SNS 자동화를 이야기할 때 빠질 수 없는 것이 `agent-browser` 스킬이다. Claude Code는 터미널에서 동작하는 코딩 에이전트이기 때문에 기본적으로 웹 브라우저를 볼 수 없다. 하지만 `agent-browser`를 설치하는 순간 상황이 달라진다.

### agent-browser의 핵심 작동 방식

agent-browser는 Vercel Labs에서 만든 브라우저 자동화 CLI 도구로, Claude Code의 스킬 시스템에 설치하면 Claude에게 브라우저를 조작하는 능력을 부여한다. 작동 방식은 다음과 같다. `agent-browser open <url>`로 페이지를 열고, `agent-browser snapshot -i`로 현재 페이지의 인터랙티브 요소들을 캡처해 `@e1`, `@e2` 같은 참조 코드를 붙인다. Claude는 이 참조 코드를 사용해 특정 버튼을 클릭하거나 폼을 채우거나 텍스트를 입력하는 동작을 수행한다.

실제로 6가지 브라우저 자동화 도구(Chrome DevTools MCP, Claude in Chrome, WebFetch, agent-browser, PinchTab, browser-use)를 수개월간 테스트한 개발자의 결론은 명확했다. SNS 자동화에는 WebFetch를 절대로 쓰지 말고 agent-browser를 쓰라는 것이다. 그 이유는 컨텍스트 효율성에 있다. Chrome DevTools MCP가 페이지당 10,000 토큰 이상을 소모하는 반면, agent-browser는 3,000~5,000 토큰 수준을 유지한다. 컨텍스트는 배터리처럼 유한한 자원이기 때문에, 같은 작업을 몇 배 더 오래 지속할 수 있느냐가 실용적 차이를 만든다.

### browserbase의 agent-browse: 스텔스와 CAPTCHA 해결

브라우저 자동화에서 자주 부딪히는 현실적인 문제가 있다. 봇 차단이다. Browserbase의 `agent-browse` 스킬은 이 문제를 정면으로 해결한다. Stagehand라는 AI 브라우저 자동화 프레임워크를 기반으로, 안티봇 스텔스 기능, CAPTCHA 자동 해결, 레지덴셜 프록시를 지원한다. 로그인된 세션이 필요한 SNS 플랫폼에서 댓글을 달거나 콘텐츠를 업로드하는 작업에 실제로 사용된다.

### 실전 SNS 자동화 파이프라인

이 두 도구를 조합하면 다음과 같은 완전한 자동화 파이프라인을 구성할 수 있다. 매일 정해진 시간에 트렌드 리서치를 수행하고 → 수집된 정보를 기반으로 콘텐츠를 생성하며 → 이미지 규격에 맞게 크롭 및 가공하고 → 브라우저를 통해 각 SNS 플랫폼에 직접 업로드하며 → 기존 포스팅에 자동으로 댓글을 달거나 반응을 남기는 것까지 전부 연속적으로 동작한다. 사람이 개입하는 지점은 최종 검수 한 번뿐이다.

## 4. 매일 정보 리서치: 웹 검색과 에이전트의 결합

Skills의 또 다른 강력한 활용 영역은 정기적인 정보 수집 자동화다. Claude Code의 헤드리스 모드를 활용하면 GitHub 이벤트, 크론 스케줄, 외부 트리거에 의해 Claude가 자동으로 실행되도록 설정할 수 있다.

예를 들어 매일 아침 7시에 특정 키워드에 관한 최신 뉴스를 수집하고, 그것을 요약해 Slack으로 보내는 자동화 스크립트를 Claude Code로 만들 수 있다. 이 과정에서 웹 검색 도구를 스킬에 통합(`type: "web_search_20250305"`)하면 Claude가 직접 실시간 정보를 검색하고, 결과를 분석하여 의미 있는 인사이트만 추출해 전달한다. 인스타그램 인사이트 지표를 구글 스프레드시트에 기록해두고, Claude가 이를 읽어 주간 보고서를 자동 생성해 슬랙으로 발송하는 파이프라인이 실제로 작동한다.

이 자동화가 단순 스크래퍼와 다른 이유는 Claude의 추론 능력이 개입하기 때문이다. 단순히 데이터를 긁어오는 것이 아니라, 수집된 데이터 중 무엇이 중요한지 판단하고, 어떤 형식으로 전달하면 가장 유용할지 결정하며, 예상치 못한 상황(페이지 구조 변경, 네트워크 오류, 예외 케이스)에 유연하게 대응한다.

## 5. 자동으로 돌아가는 실험: 병렬 에이전트 오케스트레이션

클로드 코드의 진정한 위력은 단일 에이전트가 아니라 **여러 에이전트를 동시에 병렬로 실행**할 수 있다는 점에 있다. 공식 가이드에서도 "가장 강력한 애플리케이션 중 일부는 병렬로 여러 Claude 인스턴스를 실행하는 것"이라고 명시하고 있다.

실험 자동화의 관점에서 이것이 갖는 의미는 크다. 한 Claude가 코드를 작성하는 동안 다른 Claude가 그것을 검토한다. 또 다른 Claude는 테스트를 실행하고 있다. 이 구조는 여러 엔지니어가 병렬로 작업하는 것과 동일한 효과를 낸다. Anthropic의 데이터 사이언티스트들은 실제로 Claude Code를 사용해 Jupyter 노트북을 읽고 쓰며 데이터 실험을 수행하고 있고, Claude는 이미지를 포함한 출력까지 해석할 수 있어 시각적 데이터 분석에도 활용된다.

스킬을 활용한 자동 실험 루프는 다음과 같이 설계할 수 있다. 실험 조건을 정의하는 스킬 하나, 코드를 실행하는 스킬 하나, 결과를 분석하는 스킬 하나, 그리고 다음 실험 조건을 제안하는 스킬 하나. 이 네 스킬이 맞물려 돌아가면 사람이 자리를 비운 사이에도 실험이 자율적으로 진행된다.

## 6. 자동 댓글 달기: 스킬 + 브라우저 자동화의 조합

자동 댓글은 기술적으로는 단순하지만, 실제로 유용하게 만들려면 꽤 정교한 스킬 설계가 필요하다. 단순히 댓글을 다는 것이 아니라, **어떤 포스팅에 어떤 톤으로 무슨 내용을 남길지** 판단하는 과정이 포함되어야 의미 있는 자동화가 된다.

agent-browser로 특정 해시태그나 계정의 포스팅을 순회하고, 각 포스팅의 내용을 읽어 컨텍스트를 파악한 뒤, 브랜드 목소리와 일치하는 진정성 있는 댓글을 생성해 작성하는 전체 흐름이 스킬로 구성된다. 여기서 스킬 안에 브랜드 가이드라인과 댓글 규칙(너무 홍보성으로 보이지 않을 것, 특정 단어 사용 금지, 답글 시 상대방의 이름을 포함할 것 등)을 상세히 기술해두면 Claude는 그 지침을 따라 일관된 품질의 댓글을 생성한다.

## 7. 스킬을 "깎는다"는 것의 진짜 의미

이 모든 자동화가 가능하다는 것을 알면서도 "왜 나는 잘 안 되지?"라고 느끼는 사람들이 있다. 그 차이는 대부분 스킬을 얼마나 정밀하게 설계했느냐에서 온다. 스킬을 깎는다는 것은 단순히 SKILL.md 파일을 작성하는 것이 아니다.

첫째로 **description의 정밀도**가 중요하다. Claude가 언제 이 스킬을 자동으로 호출해야 하는지를 판단하는 유일한 단서가 description이기 때문에, 트리거 조건을 구체적이고 명확하게 기술해야 한다. "SNS 관련 작업"이라고 쓰는 것과 "인스타그램 포스트 이미지 생성, 카드뉴스 제작, 플랫폼별 크기 최적화 요청 시 사용"이라고 쓰는 것은 자동 호출 정확도에서 큰 차이를 만든다.

둘째로 **워크플로우의 완결성**이다. 스킬 안에 기술된 지침이 빠진 단계 없이 처음부터 끝까지 완전한 흐름을 갖추어야 Claude가 중간에 멈추거나 사용자에게 추가 질문을 하는 일 없이 자율적으로 작업을 완수한다. Definition of Done을 스킬 안에 명시하는 것이 중요한 이유가 여기에 있다.

셋째로 **반복 사용을 통한 개선**이다. 스킬은 처음부터 완벽하지 않다. 실제로 사용해보고 Claude가 어디서 실수를 하는지, 어떤 상황에서 의도와 다른 판단을 내리는지를 관찰하며 SKILL.md를 계속 갱신해나가는 과정이 진짜 스킬 깎기다. 조직 내부에서 쌓아온 노하우가 점점 스킬 문서에 녹아들고, 그 스킬이 팀 전체의 공유 자산이 되는 순환이 만들어진다.

## 8. 스킬 생태계의 현재: 40만 개 이상의 커뮤니티 스킬

2025년 12월 앤트로픽이 Agent Skills 명세를 오픈 스탠다드로 공개한 이후, OpenAI도 Codex CLI에서 동일한 포맷을 채택했다. 스킬은 특정 도구에 종속되지 않은 **범용 AI 에이전트 확장 표준**이 되어가고 있다.

현재 SkillsMP 마켓플레이스에는 40만 개 이상의 커뮤니티 스킬이 등록되어 있으며, Claude Code, OpenAI Codex CLI, ChatGPT, Cursor 등 여러 플랫폼과 호환된다. `npx antigravity-awesome-skills` 명령어 하나로 설치할 수 있는 검증된 스킬 컬렉션도 등장했다. Frontend Design, Browser Use, Code Reviewer, Remotion, Google Workspace 연동 스킬 등 실무에서 즉시 활용 가능한 스킬들이 지속적으로 늘어나고 있다.

한국 커뮤니티에서도 카드뉴스 생성기, 웹페이지 마크다운 변환, 메타 프롬프트 생성기, DuckDuckGo 검색 스킬 등 실용적인 스킬들이 오픈소스로 공유되고 있다.

## 9. 모델의 발전을 레버리징하는 가장 쉬운 방법

마지막으로 가장 중요한 통찰을 정리한다. AI 모델은 매달 더 강력해지고 있다. 그런데 모델이 강력해질수록 가장 큰 혜택을 받는 사람은 그 모델을 잘 다루는 사람이다. 스킬을 정밀하게 설계해두면, 모델이 발전할수록 **동일한 스킬이 더 좋은 결과를 낸다**. 추가 작업 없이도 자동으로 업그레이드되는 효과를 얻는 것이다.

반대로 스킬이 없거나 조잡하게 만들어진 상태에서는 아무리 모델이 좋아져도 그 발전분을 흡수하지 못한다. 이것이 "모델의 발전을 레버리징하는 가장 쉬운 방법은 공급자들이 제공하는 도구를 최대한 활용하는 것"이라는 말의 실체다.

Skills를 열심히 깎는 사람과 그렇지 않은 사람 사이의 생산성 격차는 단기적으로는 작아 보인다. 하지만 6개월, 1년이 지나면 그 격차는 따라잡기 어려운 수준이 된다. 스킬에 쌓인 노하우가 복리로 축적되기 때문이다.

## 결론: 스킬은 AI 시대의 자본이다

Claude Skills는 단순한 자동화 도구가 아니다. 개인과 팀의 업무 노하우를 AI가 이해할 수 있는 형태로 문서화하고, 그것을 반복 재사용 가능한 자산으로 만드는 시스템이다. n8n이나 Zapier가 "A가 발생하면 B API를 호출하라"는 확정적 명령 체계라면, Claude Skills는 "C 상황에서는 D 가이드라인을 지켜서 작업하라"는 맥락 중심의 지시다. 변화하는 상황에 맞춰 유연하게 판단하고, 팀과 함께 성장하는 구조다.

SNS 자동화, 정보 리서치, 자동 실험, 자동 댓글. 이 모든 것이 Skills를 충분히 깎았을 때 가능하다는 것은 이제 수많은 사용자들의 실제 경험이 증명하고 있다. 생각하는 모든 것이 다 된다는 말은 여전히 과장처럼 들릴 수 있다. 하지만 그 말이 사실이 되는 순간은, 스킬을 충분히 깎아본 사람에게만 찾아온다.
